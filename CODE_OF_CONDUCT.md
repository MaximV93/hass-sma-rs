# Code of Conduct

hass-sma-rs is a small project with a handful of contributors. The
rules are short:

1. **Be direct + technical.** Disagree with ideas, not people. A
   sharp "this design is wrong because X" beats a vague "maybe
   consider an alternative" every time.

2. **Include reproducers.** For bugs, PRs, and reviews — raw bytes,
   exact log lines, captured fixtures are worth a thousand English
   sentences. SMA's wire protocol is weird enough that hand-waving
   doesn't help.

3. **No personal attacks, harassment, or off-topic politics.** If
   your comment would embarrass the project on a public timeline,
   rewrite it. Technical criticism doesn't need a target.

4. **Respect the maintainer's time.** hass-sma-rs is a solo
   side project. Reviews happen in evenings; expect days, not hours.
   Bumping an issue after 48h is fine; bumping after 12h is not.

5. **Attribution matters.** Credit reporters of bugs + authors of
   ideas in commit messages / PR descriptions. Don't claim work
   that's not yours.

6. **CC-BY-NC-SA boundary.** SBFspot is our protocol reference, not
   a code source. Do not copy-paste their implementation — read it,
   understand it, write clean-room Rust. This keeps our MIT/Apache
   dual-license clean.

## Enforcement

Violations should be reported to the maintainer via
<maxim.vandebroeck@gmail.com> or a private GitHub advisory. The
maintainer will:

1. Acknowledge receipt within 24h.
2. Investigate — may involve clarifying questions to both parties.
3. Act proportionally: from "please rephrase" to temporary ban
   (issue/PR commenting) to permanent block from the project's
   repositories.

Persistent bad-faith behavior is grounds for permanent removal. The
maintainer's call is final — this isn't a court.

## Why this exists

Not because there's a problem today — this is a healthy single-
maintainer project in 2026-04. Writing the CoC up-front sets
expectations before a problem appears.
